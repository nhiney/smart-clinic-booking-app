## Summary

Build a production-ready, mobile-first smart clinic booking system with appointment management, real-time queue tracking, QR-based check-in, multi-channel notifications, and medical record management. Prioritize accessibility for elderly and rural patients through large UI, voice assistant, and offline-first capability. Multi-role support (Patient, Doctor, Admin) with secure data handling (encryption, RBAC, audit logs) and idempotent payment/booking operations.

**Technical approach**: Flutter mobile app with Firebase backend, clean architecture (Presentation/Domain/Data layers), event-driven queue updates via Cloud Functions, offline-first data sync via Hive, CQRS for separating read/write operations, and React web admin dashboard for hospital staff management.

## Technical Context

**Language/Version**: Dart 3.x (Flutter), Firebase Admin SDK (Node.js 18+)
**Primary Dependencies**: Flutter 3.x, Firebase (Firestore, Auth, FCM, Cloud Functions), Provider (state management), Dio (HTTP), Hive (local storage), Firebase Admin SDK
**Storage**: Firestore (primary), Hive (local cache on mobile, offline-first)
**Testing**: Mockito (unit), Firebase Emulator (integration), Flutter integration tests (E2E)
**Target Platform**: Android 7+ & iOS 13+ (mobile-first), Web admin dashboard (React)
**Project Type**: Mobile app (Flutter) + Backend (Firebase Cloud Functions) + Web admin (React)
**Performance Goals**: App launch <2s (3G), booking completion <3 min, queue updates <5s, payment success >98%, zero double booking
**Constraints**: <2s mobile load, <500ms interaction response time (p95), offline support critical, AES-256 encryption, TLS 1.3+, WCAG AA accessibility
**Scale/Scope**: Single hospital MVP (20-50 doctors, 500-1000 bookings/day), 3+ user roles, 10+ screens, event-driven queue system

## Complexity Tracking

> **Principle VI (Idempotent Operations)**: Booking and payment mutations require idempotency keys. Firestore transactions ensure atomicity (booking + payment succeed together or fail together). Justification: Healthcare systems cannot allow double-booking or double-charging; network failures are inevitable in rural areas.

> **Principle X (Offline-First)**: Mobile app must function without internet. Hive local cache + automatic sync adds ~10-15% to schedule but is non-negotiable for rural users. Justification: Target demographic has intermittent connectivity; system must not block core workflows (view bookings, medical records).

> **Principle XI (Event-Driven)**: Cloud Functions publish domain events to Pub/Sub. Complexity: ~2 weeks additional dev for event model + subscribers. Justification: Decouples services; future notifications (SMS, email, push) can subscribe to BookingCreated without touching booking logic.
